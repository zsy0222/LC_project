## 零、行业背景：为什么前端分类比后端建厂更重要

### 0.1 核心数据

| 指标 | 数值 | 来源 |
|------|------|------|
| 全国年产厨余垃圾 | **9000 万吨+** | 住建部/行业统计 |
| 实际专门处理率 | **< 20%** | 2024 年行业数据 |
| 前端分类后杂质含量 | **≥ 25%**（塑料袋、餐盒、筷子混入） | 业内共识 |
| 厨余处理厂平均负荷率 | **< 50%** | 处于"建成即闲置"状态 |
| 焚烧厂平均负荷率 | **≈ 60%** | 严重过剩 |
| 全国处理缺口 | **1.2 万吨/日** | 2025 年预估 |

### 0.2 核心矛盾

```
不是处理厂不够——是建了也用不起来。

厨余处理厂负荷率 < 50% 的原因：
  前端分类不纯 → 杂质超 30% → 处理设备无法运转
  → 大量厨余被混入生活垃圾 → 直接焚烧/填埋

另一边：
  焚烧厂产能严重过剩（负荷率 60%）
  厨余本该去厌氧消化产沼气
  却被烧掉——既浪费资源又产生额外碳排放
```

### 0.3 关键洞察

**处理能力不足的根因在前端不在后端。**

```
前端杂质每降低 10%
  → 处理厂可用原料增加 10%
  → 现有设施负荷率从 50% → 60% → 70%
  → 不需要新建任何设施，处理能力自然提升
  → 沼气产量增加 + 焚烧量减少 = 双向减碳
```

### 0.4 虎哥模式印证

虎哥（HUGE）在杭州余杭做到了 **95% 资源化率**，核心手段就是前端分类——垃圾袋贴二维码精确溯源到户，394 个服务站覆盖 90 万户。

但虎哥有三个局限：
- 重资产模式依赖政府补贴（约 0.92 元/户/日）
- 不做厨余垃圾
- 无法复制到三四线城市和校园

### 0.5 本项目的切入点

**虎哥靠人力和重资产做前端分类，我们靠 AI 引导用户自主分类。**

适用于校园场景——外卖残余每顿都产生，宿舍楼下就有分类桶，学生有环保意愿但缺乏动力和指导。不需要建处理厂，不需要政府补贴，只需要在"吃完"和"扔掉"之间加一层 AI 引导。

## 一、为什么"熟食分类型"是错误方向

### 1.1 现实中食堂/宿舍的外卖残余长什么样

吃完一顿外卖后的真实场景：

```
桌上残留：
  🍚 剩饭（米饭粘在碗底）
  🥬 菜叶（炒青菜的残渣）
  🦴 骨头（鸡腿骨、排骨碎）
  🥡 塑料餐盒（PP 材质，透明或白色）
  🥢 一次性筷子（竹/木）
  🧻 纸巾（擦嘴用的）
  🥤 饮料杯（纸杯或塑料杯）
  🍶 酱料盒（小塑料盒）
```

AI 如果只做"熟食分类"（米饭 vs 面食 vs 肉类 vs 蔬菜），完全忽略了占体积最大的塑料餐盒、筷子、纸巾。**这恰恰是最需要分类的**——因为食物进厨余桶，塑料进回收桶，筷子纸巾各走各路。

### 1.2 结论：不应该按"食物类型"分，而应该按"回收流向"分

| 回收流向 | 实际包含物 | 去往 |
|---------|-----------|------|
| 厨余 | 剩饭、菜渣、骨头、蛋壳 | 厌氧消化 → 沼气 |
| 塑料 | 餐盒、盖子、酱料盒、饮料杯 | 塑料再生 |
| 纸/木 | 纸杯、纸巾、纸袋、筷子 | 纸浆回收 |
| 金属 | 易拉罐、锡纸 | 回炉熔炼 |

这样 AI 的职责就很清晰了：**拍一张外卖残余照片，AI 自动分流到不同回收流向**。

## 二、AI 识别怎么做

### 2.1 仍然用 ImageNet 类别映射（和纸箱系统同架构）

ImageNet 1000 类中可关联的清单：

```
厨余流向：
  米饭/谷物: rice(930), bulgur(416), mashed_potato(964)
  面食:     spaghetti(489), carbonara(497), macaroni(965)
  蔬菜:     cauliflower(101), broccoli(937), cucumber(943), bell_pepper(945)
  肉类:     meat_loaf(965), stew(745), hamburger(933), hotdog(934)
  混合:     plate(925), trifle(587), potpie(967)

塑料流向：
  塑料瓶:   water_bottle(898), pop_bottle(737)
  塑料袋:   plastic_bag(728)
  塑料容器: soap_dispenser(765) 近似映射
  泡沫:     (无直接类，用纯白/半透明区域检测兜底)

纸/木流向：
  纸板:     carton(478), packet(692)
  纸巾:     paper_towel(706)
  木材:     matchstick(695), walking_stick(855)

金属流向：
  易拉罐:   beer_bottle(440) 的金属盖区域
  铝箔:     (无直接类，靠颜色+光泽规则兜底)
```

### 2.2 多标签输出（一条记录 → 多品类）

现有架构每次只输出 1 个品类（取最高概率类别）。改为：

```
预测结果:
  品类1: 塑料 → 概率 0.78 → 对应批次 BATCH-xxx-塑料
  品类2: 厨余 → 概率 0.62 → 对应批次 BATCH-xxx-厨余
  品类3: 纸/木 → 概率 0.41 → 低于阈值 0.5，不归入

最终: 当前用户一次投递产生了 2 条批次归入
```

### 2.3 实现方式

保持 MobileNetV3 + `_imagenet_idx_to_cat` 映射，只改映射表。然后不取 `max`，取所有超过 5% 概率的类别聚合到各流向。

```python
# 原来（纸箱系统）
best = max(cat_scores, key=cat_scores.get)
# 改为（多品类）
results = []
for flow in ["厨余", "塑料", "纸/木", "金属"]:
    if cat_scores[flow] > 0.05:   # 5% 阈值
        results.append(flow)
```

### 2.4 杂质/不可回收物的处理

不再需要"拒收"逻辑。如果某个物品不属于任何回收流向（概率之和极低），标记为"未知/请手动分类"即可。用户仍然可以提交，只是 AI 无法自动分流。

## 三、与纸箱系统的对比

| | 纸箱系统 | 外卖残余系统 |
|---|---------|------------|
| AI 分类维度 | 1 个（品类） | **多标签**（多个回收流向） |
| 单次提交产生批次 | 1 个 | **1~4 个** |
| 是否需要计数 | 需要（看有几个箱子） | **不需要**——每品类计 1 次即可 |
| 是否需要完整度评估 | 需要（完好→共享，破损→回收） | **不需要**——残留量不影响处理方式 |
| `_grade_by_stats` | 边缘密度→完好/破损 | **删除或用不到** |
| 是否需要杂质检测 | 不需要（纸箱就是纸箱） | **不需要显式做**——不属于任何流向自动归为未知 |
| 拒收逻辑 | 相似度太高拒绝、冷却拒绝 | **相同，不变** |

## 七、关于"熟食分类"的回答

回到你最初的问题——AI 识别具体怎么做，不是按"米饭/面食/肉/蔬菜"分，而是按**回收流向**分：

```
照片中有什么 → AI 用 ImageNet 映射归类 →
  食物类像素多 → 标记"厨余"
  塑料类像素多 → 标记"塑料"
  木质/纸类像素多 → 标记"纸/木"
  金属光泽区域 → 标记"金属"

用户感知: "我拍了一张照，AI 自动帮我分好了三类垃圾"
```

这就回答了"具体怎么说"——**不说食物类型，说回收流向**。食物分类是手段，回收分流是目的。

## 八、两步分离设计（v3·最终版）

### 8.1 核心洞察

前面的讨论一直在纠结"怎么验证用户真的分了"——GPS 校验提前到拍照环节，导致用户必须走到分类点才能拍照，增加操作摩擦。

换个思路：**拍照和分类本就发生在不同时刻。**

```
吃完外卖（在桌前） → 拍照记录"我产生了一份厨余"
走到分类点         → GPS 确认"我到了" → 确认"我倒了"
```

两步分离，各自独立。心理上：既然都走到分类点了，顺手就倒了。

## 九、全品类分类入口设计

### 9.1 设计原则

> **不是所有品类都需要 GPS 验证。只有熟食需要，因为厨余容易腐败、需要即时处理。**

| 品类 | 分类方式 | 减碳计算 | 说明 |
|------|---------|---------|------|
| 🍚 外卖厨余 | 两步验证（拍照→到点确认） | ✅ 计算 | 吃完就拍，到点确认，每日最高频 |
| 📦 快递纸箱 | 到点→拍照→确认 | ✅ 沿用现有系数 | 已有完整减碳计算 |
| 🥡 塑料 | 到点→拍照→确认 | ❌ 暂不计 | 塑料瓶/餐盒，培养习惯为主 |
| 🔋 有害类 | 到点→拍照→确认 | ❌ 暂不计 | 电池/灯泡/药品，培养习惯为主 |

**为什么是这四个**：外卖厨余（每顿饭产生）、快递纸箱（网购拆箱）、塑料（饮料瓶餐盒）、有害（电池灯泡过期药）。覆盖校园垃圾绝大多数场景。

### 9.3 各品类流程

只有两种流程，不是六种：

```
┌──────────────────────────────────────────┐
│  流程 A：两步分离（仅外卖厨余）          │
│                                          │
│  拍照（不限地点） → pending              │
│      → 到分类点 → GPS → 勾选 → confirm  │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  流程 B：一步到位（其他五类通用）        │
│                                          │
│  到分类点 → GPS → 拍照 → AI → 勾选 → 提交 │
│                                          │
│  塑料/纸类/玻璃/有害/纸箱 完全相同       │
└──────────────────────────────────────────┘
```

| 品类 | 流程 | 减碳计算 |
|------|------|---------|
| 🍚 熟食 | 流程 A（两步分离） | ✅ 计算 |
| 🥡 塑料 | 流程 B（一步） | ❌ 暂不计 |
| 📄 纸类 | 流程 B（一步） | ❌ 暂不计 |
| 🫙 玻璃 | 流程 B（一步） | ❌ 暂不计 |
| 🔋 有害类 | 流程 B（一步） | ❌ 暂不计 |
| 📦 纸箱 | 流程 B（一步） | ✅ 沿用现有减碳系数 |

### 9.4 为什么只有熟食做两步分离

| | 熟食 | 塑料/纸/玻璃/有害 |
|---|------|-------------------|
| 产生时机 | 吃完饭立刻产生，不能等 | 可以攒一堆再扔 |
| 腐败风险 | 高温下几小时发臭 | 不会腐败 |
| 拍照时机 | 吃完就拍，记录"我刚吃完" | 到点就拍，记录"我扔了" |
| 心理逻辑 | 拍完→走到点→倒掉 | 攒够→走到点→一起扔 |

### 9.9 数据库改动

现有 7 表不变。`submissions` 加一个字段：

```

  waste_type: 熟食 | 塑料 | 纸类 | 玻璃 | 有害 | 纸箱   （新增）
```

`carbon_factors` 暂时只补熟食数据，其他品类留空（`factor` 为 0 即表示未接入减碳计算）。

### 9.10 连续打卡激励

所有品类共同计入连续打卡天数：

```
今天分类了塑料 → 连续第 1 天
今天分类了厨余 → 连续第 2 天
今天分类了纸类 → 连续第 3 天

（不限品类，只要做了一次分类就算）
```

| 连续天数 | 称号 | 奖励 |
|---------|------|------|
| 1 | 初试分类 | - |
| 3 | 分类新手 | 碳积分 ×1.1 |
| 7 | 分类达人 | 碳积分 ×1.2 + 徽章 |
| 30 | 零废弃大师 | 碳积分 ×1.5 + 专属徽章 |

断签降一级，不归零。
