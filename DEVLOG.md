# 校园低碳回收全路径追踪系统 · 开发日志

> 记录从选题到完成的完整开发过程，用于答辩展示"真实的开发经历"与制作 PPT。

---

## 一、选题阶段

### 1.1 选题来源

**痛点观察**：校园内每天产生大量快递纸箱、塑料瓶、玻璃瓶，但回收链条不透明——学生扔进回收箱后再也不知道这些东西去了哪里、是否真的被回收或再利用。这种"投完即忘"的体验让回收行为缺乏成就感和持续动力。

**选题方向**：做一个能"追踪回收物去向"的系统，让学生看到自己的回收物被做成了什么、减了多少碳。

### 1.2 选题评估

- ✅ 真实校园痛点，用户场景清晰
- ✅ 核心链路明确（扫→拍→识→追→馈），闭环完整
- ✅ Demo 可落地，不依赖外部系统
- ⚠️ AI 识别是技术难点，但可用预训练模型 + 规则策略折中
- ✘ 不追求大而全，聚焦核心闭环

### 1.3 技术选型决策

| 考量点 | 决策 | 理由 |
|--------|------|------|
| 后端框架 | FastAPI | Python 生态丰富，自动生成 API 文档，适合快速开发 |
| 数据库 | SQLite | Demo 阶段无需 MySQL，单文件零配置 |
| 前端框架 | 原生 HTML/CSS/JS | 开发周期仅 2 周，Vue/React 学习成本高，原生足够演示 |
| AI 模型 | MobileNetV3 Small | 轻量、可 CPU 运行、预训练权重可直接用 |
| AI 策略 | 预训练 + 类别映射 + 启发式规则 | 无时间采集标注数据做微调，工程折中 |
| 图片存储 | 本地文件系统 | Demo 无需 OSS |

> **核心原则**：合适比复杂更重要，先跑通再优化。

## 二、系统设计

### 2.1 核心闭环

```
用户扫码 → 拍照上传 → AI识别品类/完整度 → 自动归入批次
    → 去向端认领批次 → 上传成品照片 → 广播反馈至全部投递用户
```

### 2.2 数据库设计（7 张表）

| 表名 | 用途 | 设计要点 |
|------|------|----------|
| users | 用户 | 区分 student/reuser/admin 三种角色 |
| points | 回收点位 | 每个点位有固定二维码标识 |
| batches | 批次（核心） | 按点位+日期+品类+破损等级自动归集，批次号包含完整信息 |
| submissions | 投递记录 | 关联用户和批次，记录 AI 识别结果与减碳量 |
| reuse_items | 成品反馈 | 去向端上传的再生品照片与描述 |
| notifications | 通知 | 批次完成后向投递用户推送 |
| carbon_factors | 碳因子 | 环保专业数据，品类×路径的减碳系数 |

**设计思想**：以批次（batches）为骨架串联整个链路——批次号 `BATCH-{点位}-{日期}-{品类}-{路径}` 本身就能表达完整信息。

### 2.3 批次归集逻辑

不同破损等级的同类回收物会进入不同批次：
- 完好 → A 路径（共享/再利用）
- 轻损 → B 路径（手工改造）
- 破损/受潮 → C 路径（直接回收）

这样去向端可以按等级精准筛选，而不是混在一起。

## 三、AI 的帮助

### 3.1 AI 辅助开发概览

本项目在开发过程中使用了 Claude Code 作为 AI 编程助手，以下是 AI 参与的具体环节：

| 环节 | AI 角色 | 具体贡献 |
|------|---------|----------|
| 项目初始化 | 代码生成 | 生成 FastAPI 项目骨架、ORM 模型、API 路由模板 |
| 数据库设计 | 方案建议 | 建议以批次号作为核心串联 ID，含点位/日期/品类信息 |
| AI 识别模块 | 算法实现 | 实现 MobileNetV3 加载、ImageNet 类别映射聚合、图像统计启发式规则 |
| 前端 SPA | 代码生成 | 生成完整单页应用，含登录/投递/去向/排行/故事时间轴 |
| 碳因子计算 | 业务逻辑 | 实现查表计算与兜底默认值 |
| 通知广播 | 业务逻辑 | 实现批次完成后向全部投递用户推送 |
| 文档更新 | 文档撰写 | 根据实际代码更新 README 和设计文档，消除文档与代码不一致 |
| 开发日志 | 文档撰写 | 协助整理本文档，结构化记录开发过程 |

| 设计参数集中管理 | 方案执行 | 将散落各处的魔法数字（200m、30s、0.15）统一迁入 config.py + DESIGN_VALUES.md，实现"改一处全局生效" |

### 3.2 AI 使用心得

**AI 擅长的**：
- 快速搭建项目骨架和样板代码
- 根据需求生成符合规范的 API 接口
- 处理繁琐的前端 HTML/CSS/JS 布局
- 文档整理和格式化

**AI 不擅长的 / 需要人工判断的**：
- AI 识别到底用微调还是预训练——需要根据时间/数据约束做决策
- 批次归集的粒度（按等级分还是合并）——需要理解业务逻辑
- 前端交互设计（弹窗、Tab 切换、上传流程）——需要从用户视角思考

**关键经验**：AI 是"加速器"而非"自动驾驶"——它能快速生成代码，但架构决策、业务逻辑、技术折中仍需人工判断。

### 3.3 重要发现：AI 辅助开发中的人类主导时刻

在完成反作弊功能（GPS 校验 + 冷却 + 图片去重 + 多物品计数）后，我发现代码中出现了大量散落的"魔法数字"——`200` 米、`30` 秒、`0.15` 相似度阈值——分散在 `submission.py`、`index.html` 的不同位置。如果后续需要调参（比如把冷却时间改成 60 秒），需要翻遍多个文件逐一修改，极易遗漏。

于是我想到：**应该有一个统一的设计参数表，把所有可调数值集中在一个地方**。

我让 AI 执行了以下操作：
1. 创建 `DESIGN_VALUES.md`——人类可读的完整参数字典，含公式说明和数据来源
2. 将散落在 `submission.py` 中的硬编码常量迁移到 `src/config.py` 统一管理
3. 更新 `submission.py` 从 `config` import，消除所有魔法数字

**这个环节的意义**：

这不是 AI 主动提出的建议——AI 只是按我的要求生成了反作弊代码，魔法数字散落是它默认的行为。**发现这个问题、意识到需要集中管理、提出解决方案，是我作为开发者的判断**。AI 的角色是高效执行——它在一分钟内完成了参数提取、文档生成、代码迁移的全部工作。

**这个模式值得记录**：在 AI 辅助开发中，人类的核心价值在于"发现问题"和"提出正确的抽象方向"，AI 的价值在于"快速执行"。两者分工明确——人类做架构决策，AI 做代码落地。

这也印证了何棋在分享中提到的观点：**AI 擅长创造与探索，但在执行确定性任务时存在不可靠性**。反过来说，AI 在确定性执行上可靠（改配置、写文档），但在需要"大局观"的事情上（比如发现魔法数字散落的设计问题），仍需人的介入。

## 四、遇到的问题与解决方案

### 问题 1：AI 模型选择——微调还是直接用？

**现象**：设计文档最初写的是"MobileNetV3 微调"，但实际没有回收物标注数据集。

**分析**：
- 采集和标注数据至少需要 3-5 天
- 微调后的模型可能过拟合小数据集
- Demo 演示只需要能区分纸箱/塑料/玻璃即可

**解决方案**：
1. 使用 ImageNet 预训练权重，找出与三品类相关的类别索引
2. 将相关类别的概率聚合，得到品类得分
3. 结合图像统计特征（边缘密度、亮度方差）评估完整度
4. 总分过低时拒绝识别，提示重拍
5. 增加 mock 降级模式，确保无 torch 环境也能演示

**反思**：这是典型的"工程折中"——时间不够时，用规则策略替代数据驱动，保证 Demo 可演示。答辩时可以说清楚：当前方案是 MVP，后续有数据后可以微调提升。

---

### 问题 2：前端技术选型——Vue 还是原生 JS？

**现象**：设计文档写"Vue3"，但实际用了原生 HTML/CSS/JS。

**分析**：
- 开发周期仅 2 周，还要兼顾期末
- 团队可能没有 Vue 经验
- Demo 只需 3 个 Tab 页面 + 几个弹窗，交互不复杂

**解决方案**：用原生 JS 写单页应用（SPA），一个 HTML 文件包含全部功能：
- 登录选择 → 主界面三 Tab（投递/去向/排行）
- 弹窗（认领/成品上传/批次故事）
- Toast 提示

**反思**：原生 JS 在 Demo 阶段完全够用，代码量约 600 行，维护成本低。如果后续需要更复杂交互再迁移到 Vue。

---

### 问题 3：文档与代码不一致

**现象**：设计文档写"微调"实际没微调，写"Vue3"实际是原生 JS，写"reuser.html / rank.html 独立页面"实际是重定向。

**影响**：如果评委对比文档和代码发现不一致，会质疑项目真实性。

**解决方案**：
1. 逐项核对文档与代码
2. 更新 README.md：启动页面、AI 说明、目录结构
3. 更新设计文档.docx：系统架构、接口表（8→15）、技术栈表、开发计划
4. 原则：**文档如实反映代码，不夸大**

---

### 问题 4：批次归集的粒度设计

**现象**：同一品类但不同破损程度的回收物，去向不同（完好的可以共享，破损的只能回收），如果全部混在一个批次里，去向端无法区分。

**解决方案**：批次号增加路径后缀 `-{path}`，不同破损等级自动归入不同批次。
- `BATCH-A03-20260528-纸箱-A`（完好，可共享）
- `BATCH-A03-20260528-纸箱-C`（破损，直接回收）

---

### 问题 5：AI 不可用时的兜底

**现象**：torch 体积大（~800MB），安装可能失败；无 GPU 的机器上推理慢。

**解决方案**：
1. Mock 模式：基于 RGB 通道统计的简单规则分类
2. 启动时自动检测 torch 是否可用，不可用则降级
3. 前端展示当前 AI 模式（torch / mock）
4. 所有 AI 输出都带置信度，低置信度提示用户重拍

### 问题 6：用户注销后重新登录，信息重复显示

**现象**：退出登录后再次选择用户进入，顶栏的昵称、角色、碳积分信息出现重复/叠加。

**根因**：`initLogin()` 每次被调用时，向 `<select>` 元素不断 `appendChild` 新的 `<option>`，但没有先清空已有选项。退出→重新登录时会再次调用 `initLogin()`。

**解决方案**：在 `initLogin()` 中填充用户列表前，先执行 `sel.innerHTML = '<option value="">-- 选择用户 --</option>'` 清空已有选项。

### 问题 7：缺少定位校验——用户可以不在回收点就提交

**现象**：用户在任何地方点击选择回收点即可投递，存在虚假投递的可能。

**解决方案**：
1. 点位数据已有 GPS 坐标（`gps` 字段），前端 `loadPoints` 时解析 `lat,lng`
2. 选择点位时调用浏览器 `navigator.geolocation.getCurrentPosition()` 获取用户位置
3. 使用 Haversine 公式计算用户与点位距离
4. 距离 ≤ 200 米 → 弹出 toast "✅ 位置验证通过，可进行投递"
5. 距离 > 200 米 → 提示用户到达点位后再投递，提交按钮禁用
6. 定位失败/浏览器不支持 → 降级放行（弹警告但不阻塞）

---

### 问题 8：防止用户刷碳减排

**现象**：存在三种潜在作弊方式——① 同一点位反复提交 ② 同一个箱子换角度拍多次 ③ 无法体现多件物品同时投递。

**解决方案（三层防护）**：

**8a. 30 秒冷却机制**
- 后端：`POST /submission` 检查用户在同点位 30 秒内是否有提交记录，有则返回 429
- 前端：调用 `GET /submission/cooldown` 查询冷却剩余时间，显示倒计时条
- 冷却期间提交按钮禁用，显示 "⏳ 冷却中 (Xs)"

**8b. 图片相似度去重**
- 新增 `POST /photo/hash` 端点：计算上传图片的感知哈希（aHash）
- 算法：缩放至 32×32 灰度图 → 与像素均值比较 → 生成 1024-bit 哈希
- 提交时与用户最近 10 条提交的哈希做汉明距离比较
- 相似度 > 85%（差异 < 15%）→ 判定为同一物品，拒绝提交

**8c. 多物品计数**
- 前端新增"回收物数量"输入框（默认 1，上限 20）
- 后端 `item_count` 字段记录，碳减排 = 单件减碳量 × 数量
- 数据模型：`Submission` 表新增 `photo_hash` 和 `item_count` 字段

---

### 问题 9：AI 识别是否可以调用现有云服务 API？

**背景**：在优化 AI 纸箱计数功能时，思考是否可以接入阿里云视觉智能、百度 EasyDL 等现成 API 来替代本地模型。

**分析**：
- 技术上完全可行——`predict_image(image_bytes)` 是独立接口，替换内部实现即可，前后端无需改动
- 但 Demo 阶段接入外部 API 的风险：
  - 需要网络 → 答辩现场网络不可控
  - 需要 API Key → 涉及费用和密钥管理
  - 违反备赛原则"绝对不要把演示命运交给现场网络和服务器"

**决策**：Demo 阶段保持本地 MobileNetV3 + mock 降级方案，确保离线可演示。后续版本可考虑接入云 API 提升识别精度。

**反思**：不是所有"更好的技术"都适合当前阶段。Demo 的核心目标是稳定可演示，而非技术先进性。

### 问题 10：GPS 坐标不准——高德地图坐标 ≠ 现场实测坐标

**现象**：高德地图上选取的"南京大学鼓楼校区图书馆"坐标为 `32.054463, 118.780906`，但拿着手机站在图书馆门口实测为 `32.056447, 118.775293`，偏差约 570 米。

**根因**：
- 高德地图 POI 坐标是区域中心点，不是建筑门口精确位置
- GCJ-02 坐标系本身有偏移
- 不同设备/地图服务商的数据源不一致

**解决方案**：
1. 在页面中加入 GPS 调试面板——选择点位后实时显示：你的位置、点位坐标、距离
2. 带着手机到三个点位现场，逐个实测并更新坐标
3. 已更新：图书馆 → `32.056447, 118.775293`（现场实测）
4. 待更新：软件学院、南园3舍（需到现场后实测）

**反思**：在涉及 GPS 定位的应用中，**现场实测 > 地图坐标**。高德/百度地图的 POI 数据通勤导航够用，但 100m 级别的精确校验必须实地采集。

### v0.4 — 设计转折：从纸箱回收到外卖残余全品类分类

#### 背景

根据建议反馈，纸箱单品类回收在实际校园场景中存在两个问题：
1. 快递纸箱数量有限，且已有成熟的废品回收链
2. 校园最大体量的垃圾是外卖残余——每顿都产生，几乎没有专门回收

因此决定将项目定位从"纸箱回收追踪"转向**"外卖残余全品类分类"**。以下是完整的设计讨论过程，记录在 `WASTE_DESIGN.md`。

#### 设计迭代过程

**第一轮：熟食分类型（被推翻）**

最初设想 AI 识别外卖残余中的食物类型（米饭/面食/肉类/蔬菜），按主要成分归入批次。

问题：忽略了占体积最大的塑料餐盒、筷子、纸巾。而且"米饭 vs 面食"的分类在实际回收中没有分流意义——它们都进厨余桶。

**第二轮：按回收流向分 + AI 多标签（被简化）**

改为按"回收流向"分类（厨余/塑料/纸木/金属），AI 一张照片拆成多品类。技术可行性没问题（MobileNetV3 多标签输出），但存在一个根本矛盾：

> 怎么判断用户拍了照之后真的做了分类？

纸箱不存在这个问题（拍了=投了），但外卖残余需要用户动手把不同品类分开投——这一步系统不可观测。

**第三轮：多步打卡 + 逐一验证（被否决）**

设想让用户逐一拍照投递（先拍厨余→倒→打卡，再拍餐盒→扔→打卡），每次单独 GPS 校验。

问题：流程太复杂，用户吃一顿外卖要操作 3-4 次。操作摩擦抵消了激励效果。

**第四轮：两步分离（最终方案）**

关键洞察——拍照和分类本就发生在不同时刻。既然分离了，就只有易腐败的熟食需要"提前拍"。

> 拍照不限地点（吃完就拍）→ 走到分类点 → GPS 确认 → 勾选确认 → 提交

心理锚定效应：用户拍了照、走到了分类点，"来都来了"——自然就倒了。

#### 全品类入口设计

进入首页后可以选择分类类型。只有两种流程，不是六种：

| 品类 | 流程 | 减碳计算 |
|------|------|---------|
| 🍚 熟食 | 流程A（两步分离） | ✅ 计算 |
| 🥡 塑料 | 流程B（一步） | ❌ 暂不计 |
| 📄 纸类 | 流程B（一步） | ❌ 暂不计 |
| 🫙 玻璃 | 流程B（一步） | ❌ 暂不计 |
| 🔋 有害类 | 流程B（一步） | ❌ 暂不计 |
| 📦 纸箱 | 流程B（一步） | ✅ 沿用现有减碳系数 |

纸箱不特殊对待——和塑料、纸类、玻璃、有害完全相同的流程 B。不是六个品类六个页面，而是一个页面两种模式。

设计原则：
- 熟食是唯一走两步的品类（腐败风险需要提前拍）
- 其余五类统一：到点→GPS→拍照→勾选→提交
- 纸箱保留现有减碳计算，塑料/纸类/玻璃/有害暂不计入
- 所有品类共同计入连续打卡天数，不限品类
- 断签降级不归零

#### 设计收获

这次设计讨论长达数轮，每一次推翻都更接近真实场景：

1. "熟食分类型" → 发现忽略了非食物垃圾
2. "按回收流向分" → 发现无法验证用户行为
3. "多步打卡" → 发现流程太复杂
4. "两步分离" → 利用心理锚定而非强制验证
5. "全品类入口" → 熟食是重点，其他品类轻量接入
6. "纸箱不特殊对待" → 塑料/纸类/玻璃/有害/纸箱统一流程B，六个品类两种模式

**核心经验**：产品设计不是一步到位。发现一个问题 → 提出方案 → 发现方案的新问题 → 修正。AI 辅助设计时，人的作用是"发现方案的问题"——AI 不会主动质疑你的设计。

#### 行业调研：为什么前端分类比后端建厂更重要

调研了竞品"虎哥HUGE"（浙江，覆盖 90 万户）和中国厨余垃圾处理行业现状，得出核心结论：

> **厨余处理能力不足的根因在前端不在后端。** 全国年产厨余 9000 万吨+，专门处理率不足 20%。但厨余处理厂平均负荷率不到 50%——不是处理厂不够，是前端分类太差、杂质超 30%，导致建好的处理厂无干净原料可用。前端杂质每降低 10%，现有设施处理能力自然提升 10%，不需要新建任何设施。

虎哥靠 394 个服务站 + 政府重补贴做到 95% 资源化率，但不做厨余，且模式无法复制到校园。

**本项目的定位**：虎哥靠人力和重资产做前端分类，我们靠 AI 引导用户自主分类。适用于校园场景，轻量、可复制、不需要政府背书。

## 八、反作弊体系强化（2026-06-03）

### 8.1 背景
原有反作弊仅 3 层（GPS 定位 + 30s 冷却 + 同用户图片去重），多轮审查后发现以下漏洞：
- 同一张照片在不同天重复用 → 绕过 24h 窗口
- 多人共用同一张照片（代投）→ 跨用户无检测
- 从相册选旧图/网图 → 无拍摄时间校验
- 单用户无上限 → 可刷分

### 8.2 新增 4 层防线

| # | 防线 | 触发条件 | 管理员 |
|---|---|---|---|
| 4 | 每日投递上限 | 同日 confirmed+pending ≥20 次 | 跳过 |
| 5 | 同点长周期去重 | 同用户+同点位 7d 内相似度 >85% | 跳过 |
| 6 | 跨用户去重 | 同点位 3min 内不同用户相同照片 | 跳过 |
| 7 | EXIF 拍摄时间 | 无 EXIF 或过期→拒绝（厨余 1h，其他 5min） | Demo/管理员跳过 |

### 8.3 关键设计决策

**跨用户窗口为什么只有 3 分钟？**
不同人投相似纸箱很正常，不能误杀。只拦截"A 拍完当场传给 B 用"的情况——这种作弊一定是同一时间同一地点发生的。

**EXIF 为什么按品类区别？**
厨余是两步流程（先拍后确认），拍到确认之间可能隔很久，放宽到 1 小时。纸箱/塑料/有害必须现场拍（5min）。

**为什么不在 upload 接口校验 EXIF？**
上传时还不知道品类，必须等提交流程拿到 `waste_type` 才能判断用哪个阈值。

**本地测试怎么办？**
`DEMO_MODE=True` 或管理员身份跳过所有 EXIF 校验。电脑上的截图/网图通常无 EXIF，Demo 模式直接放行。

### 8.4 涉及文件

| 文件 | 改动 |
|------|------|
| `src/config.py` | +3 参数（同点天数/跨用户分钟/EXIF分钟/厨余EXIF/每日上限） |
| `src/services/ai_service.py` | +`extract_exif_datetime()` + `is_exif_too_old()` |
| `src/api/submission.py` | 重构去重为 4a/4b/4c 三子校验，+每日上限，+EXIF 校验，confirm 也补去重 |
| `src/api/predict.py` | 回退，EXIF 校验移至 submission |
| `src/web/index.html` | `<input capture="environment">` 强制调系统相机 |

### 8.5 现存已知缺口

- **品类谎报**：AI 已识别品类但未与用户声称比对（低优先级，AI 本身就有误差）
- **GPS 伪造**：浏览器 F12 可改定位，服务端无法根治（需 IP 粗略校验，Demo 阶段不做）
- **多账号轮替**：需设备指纹/IP 限流（Demo 阶段不做）
- **翻拍屏幕**：摩尔纹检测误杀率高，暂不引入

## 十七、AI 模型升级：从规则策略到微调模型（2026-06-09）

### 17.1 背景

最初的 AI 识别方案是"预训练权重 + ImageNet 类别映射 + 启发式规则"，准确率有限。经过数据采集和训练，升级为真正的四分类微调模型。

### 17.2 训练成果

| 指标 | 数值 |
|------|------|
| 模型 | MobileNetV3 Small |
| 类别 | 外卖厨余 / 快递纸箱 / 塑料 / 有害 |
| 准确率 | 96.5% |
| 训练增强 | 数据增强 + 类别权重 + Label Smoothing |

## 十九、关键经验总结

### 19.1 AI 辅助开发的真实模式

经过两周密集开发，总结出 AI（Claude Code）辅助开发的真实工作模式：

**AI 擅长的**：
- 快速生成样板代码和重复性工作
- CSS 样式调整（弹窗美化、布局修复）
- 数据库迁移脚本
- 文档整理

**AI 不擅长的**：
- 理解业务逻辑的微妙之处（如"我的作品"到底该显示什么）
- 一次性做对复杂功能（往往需要 3-4 轮迭代修复）
- 主动发现设计缺陷

**关键模式**：人发现问题 → 人提出方案 → AI 执行 → 人验证 → AI 修复。循环往复。

### 19.2 `__pycache__` 陷阱

多次遇到"改了代码但不生效"的问题，根因是 Python 的 `.pyc` 字节码缓存。git reset 后旧缓存仍然存在，导致服务运行的是旧代码。

**教训**：切换分支/回退版本后，务必删除 `__pycache__` 并重启 Python 进程。

### 19.3 Demo 优先原则

在多个技术决策中（本地模型 vs 云 API、原生 JS vs Vue、SQLite vs MySQL），始终选择"Demo 可稳定演示"的方案而非"技术更先进"的方案。这在答辩场景中至关重要。
